# baseline 与 RAG 代码审查对比实验

## 1. 实验目标

实验检验外部知识检索能否提高预定义问题的召回率、减少无依据断言，并为审查意见提供可追溯
证据。对照组为纯 LLM，实验组在相同审查链上增加 FAISS Top-5 证据。

本实验不使用“准确率”描述 `命中标注问题数 / 标注问题总数`，而使用更准确的“问题召回率”。
评价协议见 [evaluation_protocol.md](evaluation_protocol.md)。

## 2. 实验设置

| 项目 | 设置 |
|---|---|
| 模型 | `deepseek-chat` |
| 温度 | `0` |
| Prompt 版本 | `2026-07-21.1` |
| 样本数 | 5 |
| 标注根因数 | 6 |
| 检索 | 通用多意图查询 + TF-IDF + FAISS，Top-5 |
| 知识库 | PEP 8、OWASP SQL 注入防护、本项目设计约束 |
| 输出 | Pydantic 结构化 JSON |
| 人工评审者 | 1 |
| Rubric | 2.0 |

5 个合成样本覆盖命名规范、空值访问、SQL 注入、HTTP/Repository 分层约束和异常吞没。
`ground_truth.json` 在模型运行前完成。两种模式共用模型、输入、基础 Prompt、结构化 schema、
质量审计和确定性校验，区别仅为 RAG 模式获得检索原文。

## 3. 评价方法

- **问题召回率**：命中的唯一 ground-truth 根因数 / 全部 ground-truth 根因数。
- **误报**：无法由 diff 或提供文档支持，或依赖未知外部上下文的意见。
- **重复意见**：已经由另一条意见覆盖的同根因重复输出。
- **建议质量**：只评价已命中真实问题的修复建议；未命中时留空。
- **整体审查质量**：综合召回、误报、重复、严重度、定位、建议和引用的 1-5 分。
- **严重度准确率**：严重度与 ground truth 完全一致的命中根因比例。
- **精确定位率**：模型定位与 ground truth 规范根因行完全一致的命中根因比例。
- **无依据断言率**：缺少代码或文档依据的意见 / 全部最终意见。
- **引用原文一致率**：source、section、chunk_id 和 quote 均通过程序原文校验的比例。
- **引用支持率**：人工判断引用内容确实支持对应结论的比例。

聚合程序会交叉验证 CSV、ground truth 和结果 JSON，避免只依赖人工手填计数。

## 4. 检索与引用验证

主实验 6 组期望证据全部进入对应 Top-5，evidence recall@5 为 **100%（6/6）**。另有 5 个
表面写法不同的检索 validation 样本：单向量查询为 **80%（4/5）**，多意图融合后为
**100%（5/5）**。validation 参与了流程改进，因此不作为最终未见测试集。

RAG 最终输出包含 6 条引用：

1. 6/6 的 source、section、chunk_id 和 quote 与本轮检索原文一致；
2. 人工复核认为 6/6 引用能够支持对应审查结论。

这两个指标分别报告，避免把“引用存在且原文匹配”误当成“引用必然支持结论”。

## 5. 逐案例结果

| 样本 | baseline | RAG | 主要差异 |
|---|---|---|---|
| 命名规范 | 未报告 | 命中函数命名根因 | RAG 引用 PEP 8，但建议未覆盖参数与局部变量 |
| 空值访问 | 命中 | 命中并引用项目规则 | 两者定位均为 11-12 行而非精确第 12 行；RAG 将 high 错标为 critical |
| SQL 注入 | 命中 | 命中并引用 OWASP | baseline 建议更贴合 SQLAlchemy；RAG 使用通用占位符表述 |
| 分层约束 | 命中空值，漏掉分层；误报缺少鉴权 | 两个根因均命中 | RAG 架构严重度偏高，两个定位均未与规范根因行完全一致 |
| 异常吞没 | 命中 | 命中并引用项目规则 | 两种模式都把同一根因拆成两条意见 |

baseline 在分层样本中报告“端点缺少鉴权”。diff 没有给出项目认证策略，因此该意见被判为依赖
未知上下文的误报。RAG 没有保留该意见，而是报告文档中明确存在的 HTTP 层边界约束。

## 6. 聚合结果

| 指标 | baseline | RAG | 解释 |
|---|---:|---:|---|
| 问题召回率 | 66.67%（4/6） | 100.00%（6/6） | RAG 多检出命名和分层根因 |
| 误报数 | 1 | 0 | RAG 消除未知鉴权策略推测 |
| 重复意见数 | 1 | 1 | 两者均重复拆分异常吞没根因 |
| 整体审查质量 | 3.2 | 3.8 | RAG 整体更完整，但仍有严重度和定位问题 |
| 已命中问题的建议质量 | 4.75（n=4） | 4.4（n=5） | RAG 未提高建议本身质量 |
| 严重度准确率 | 100.00%（4/4） | 66.67%（4/6） | RAG 两个问题严重度偏高 |
| 精确定位率 | 25.00%（1/4） | 33.33%（2/6） | 两种模式均较弱 |
| 无依据断言率 | 16.67%（1/6） | 0.00%（0/7） | RAG 输出均有代码或文档依据 |
| 引用原文一致率 | 不适用 | 100.00%（6/6） | 程序验证 |
| 引用支持率 | 不适用 | 100.00%（6/6） | 人工验证 |

结论应限定为：在当前 5 个受控样本中，RAG 提高了预定义问题召回率、减少无依据推断并提供
可追溯证据；它没有证明建议质量提升，也没有解决严重度校准、精确定位和根因去重问题。

## 7. 失败模式与工程处理

### 7.1 严重度校准

RAG 将普通空值访问标为 `critical`，并将项目分层违规标为 `high`；ground truth 分别为
`high` 和 `medium`。这说明检索证据能帮助发现问题，但不保证模型正确理解严重度量表。

### 7.2 精确定位

程序内置位置检查只验证 `line_start` 是否指向新增行。Rubric 2.0 的 `location_exact` 更严格：
必须锚定到规范根因行，且不无必要地扩展范围。两种模式在该项均表现较弱。

### 7.3 根因去重

异常吞没样本中，“静默丢弃异常”和“仍返回成功”是同一根因的机制与后果。两种模式都输出
两条意见，因此分别记录一条 `duplicate_findings`，不再让第二条处于既非真阳性也非误报的
统计灰区。

### 7.4 结构合法不等于语义正确

模型草稿曾把合法 snake_case 标识符判为 mixedCase。流水线使用同模型质量审计、确定性命名
检查和 `raw_attempts` 留痕降低风险，但这些措施无法覆盖所有语义矛盾。

## 8. 局限性

1. 样本量小且为合成 Flask/Python diff，结论不代表真实大型 PR。
2. 只使用一个服务端模型别名和一次运行，未分析随机方差和模型间差异。
3. 当前只有一名评审者，没有评审者间一致性指标。
4. ground truth 只覆盖预先选择的缺陷类型，可能遗漏其他合理问题。
5. 建议质量和整体质量仍含人工判断，虽有锚点但不是完全客观指标。
6. TF-IDF 偏向词面匹配；跨语言和复杂同义改写需要进一步验证。
7. validation 已参与检索流程改进，尚缺来自真实仓库的最终未见测试集。
8. 服务端模型别名未来可能变化，完整复现还需要供应商提供固定模型版本。

## 9. 可复现命令

```bash
python agent.py index
python agent.py audit-retrieval
python agent.py audit-retrieval \
  --samples-dir experiment/retrieval_validation/samples \
  --ground-truth experiment/retrieval_validation/ground_truth.json \
  --output experiment/results/retrieval/validation_audit.json
python agent.py experiment --mode both
python agent.py evaluate
python -m pytest -q
```

原始输出位于 `results/baseline/` 与 `results/rag/`。每个 JSON 保存输入摘要、完整 Prompt、模型
原始响应、最终意见、检索片段和全部校验记录。
